# Local LLM eval — `tasks -p` agent path (2026-07-02)

> **Historical result:** This evaluation predates the July 8 migration from Org
> files to JSONL. The sandbox names and environment variables below describe the
> system that was tested. The current equivalents are `tasks.jsonl`,
> `TASKS_FILE`, and `TASKS_ARCHIVE`. Re-run the harness before treating these
> scores as evidence for the current store.

**Question:** can a local model (behind the Hermes harness) replace Sonnet as the
default for `tasks -p` / the TUI agent, or at least beat the current Hermes
default `gemma4:e4b`?

**Method:** each candidate drives the *real* CLI (`bin/tasks -p --provider
hermes --model <M>`) against a fresh sandbox `gtd.org`, once per prompt across 8
task dimensions × 3 trials. Outcomes are scored by the repo's own parser
(`Tasks::Store`) plus a corruption check (`tasks check`). Harness:
`eval/llm/harness.rb`; raw data: `runs-initial.jsonl` (fresh-machine screening),
`runs.jsonl` (authoritative a3b). No model touched real data — sandboxes are
pinned via `TASKS_ORG`/`TASKS_ARCHIVE`.

## Verdict

**Keep `claude-cli:sonnet` as the default.** No local model is *promising enough
to default to* — the blocker is latency, not capability. The best local model,
`qwen3.6:35b-a3b`, is genuinely capable (handles all 8 dimensions, never
corrupts the file) but takes **~2 minutes per request on a good day and times
out (>4.5 min) under sustained load** — too slow and too variable for the
default "type a request, wait" experience, where Sonnet answers in seconds.

**But two concrete wins ship from this eval:**
1. `qwen3.6:35b-a3b` replaces `gemma4:e4b` as the **default *Hermes* model** — if
   you opt into the local harness (`--provider hermes` or the TUI switcher), you
   now get the one local model that actually works, not the one that derails.
2. An `AGENTS.md` guardrail (counter the "CLI is broken" hallucination; always
   use the CLI for dates) that measurably improved local-model reliability and
   is harmless-to-helpful for Sonnet too.

0 corruptions across every run. The date dimensions initially failed for two
reasons, both fixed: (a) the eval sandboxed under `/var/folders` (macOS temp),
which Hermes' file-write filter rejects as a "sensitive system path," silently
blocking direct edits — moved to `$HOME`; (b) the model distrusted the CLI and
hand-wrote `DEADLINE: 2026-07-05` *without* the required `<angle brackets>` (the
parser ignores it) — fixed by the `AGENTS.md` guardrail, after which both date
dimensions went 3/3.

## Why not default to it

- **Latency.** 118 s median at best; a later authoritative run (machine warm
  after hours of inference) saw most tasks hit the **280 s timeout**. So per
  request expect ~2–4½ minutes, highly variable. That's a poor default when
  Sonnet is seconds.
- **Prompt-sensitivity.** Reliability moved with small prompt changes (6/8 vs
  5/8 across runs) — it works, but it isn't rock-solid.
- **Safety was acceptable in this run.** It never corrupted the Org file under
  test. This result does not validate the current JSONL store.

## Recommendation

- **Default stays `claude-cli:sonnet`.**
- **`qwen3.6:35b-a3b` becomes the default Hermes model** (was `gemma4:e4b`), so
  the local/offline path uses the one model that works. Opt in with
  `llm_provider = hermes` (or the TUI `M` switcher), accepting the latency.
- **Re-evaluate** when a faster capable local model lands (a well-tuned coder
  model, or a quantization that keeps a3b's competence at lower latency). The
  harness is reusable: `ruby eval/llm/harness.rb` (env: `EVAL_MODELS`,
  `EVAL_TASKS`, `EVAL_TRIALS`, `EVAL_TIMEOUT`).
